### 🆕 NC1 · JWT 签名验证失败被静默忽略 → 伪造 token 可绕过认证

**文件**: `backend/src/main/java/com/smartenergy/backend/filter/JwtAuthenticationFilter.java`  
**行号**: 51-82  
**严重度**: 🔴 CRITICAL — 严重安全漏洞,Peer review 发现,v2 已 read-only 复核

**当前代码**:
```java
if (StringUtils.hasText(token)) {
    try {
        if (JWTUtil.verify(token, jwtConfig.getKeyBytes())) {       // ← 返回 false 时整个 if 跳过
            JWT jwt = JWTUtil.parseToken(token);
            ...
            // 校验过期、session、设置 SecurityContext
        }
        // ← 这里没有 else 分支!verify==false 时直接 fall-through 到 filterChain.doFilter()
    } catch (Exception e) {
        // 签名错误 / 格式异常 / 解析失败 → 401
        response.sendError(HttpServletResponse.SC_UNAUTHORIZED, "Token 无效");
        return;
    }
}
filterChain.doFilter(request, response);    // ← 伪造 token (verify==false) 走到这里, 请求继续
```

**注**: catch 块只捕获**抛异常**的情况(如解析失败),但 `JWTUtil.verify()` 对签名错误**不抛异常,只返回 false**,此路径被完全遗漏。

**失败场景**: 攻击者构造 `header.payload.伪造签名` → `verify()` 返回 false → 请求未经认证继续走 → 后续 controller 取 `SecurityContext` 拿到的是匿名用户,某些公开端点之外的接口可能被错误放行。

**修复方向**:
```java
if (JWTUtil.verify(token, jwtConfig.getKeyBytes())) {
    // ... existing logic
} else {
    response.sendError(HttpServletResponse.SC_UNAUTHORIZED, "Token 签名无效");
    return;
}
```

### 🆕 NC3 · SQL 注入风险点(`.last("LIMIT " + var)` 拼接)

**文件 + 行**:
- `service/impl/SensorDataServiceImpl.java:91` `.last("LIMIT " + limit)`
- `service/impl/SparePartServiceImpl.java:151` `.last("LIMIT " + Math.max(1, limit))`
- `service/impl/WorkOrderServiceImpl.java:229` `.last("LIMIT " + limit)`

**严重度**: 🔴 CRITICAL(潜在)

**说明**: MyBatis-Plus 的 `.last()` 把传入字符串原样拼到 SQL 末尾,不做转义。**当前 `limit` 是 `int` 类型由 Java 静态类型系统兜底**(无法注入字符串),但:
1. 任何 refactor 改成 `String limit`(如从 `@RequestParam` 直接拿)立刻成为利用面
2. 缺乏防御编程,违反"永不信任输入"原则
3. 同库另外 7 处 `.last("LIMIT 1")` / `.last("LIMIT 50")` 是常量安全,但这 3 处变量拼接需要立即收敛

**失败场景**: 未来某次 refactor 改成 `@RequestParam String limit` → 攻击者发 `?limit=1; DROP TABLE work_order;--` → SQL 注入成功。

**修复方向**:
- 用 MyBatis-Plus 的 `Page<T>` 分页 API 替代 `.last("LIMIT ...")`
- 或在 `.last()` 前钳制 `int safe = Math.min(Math.max(1, limit), 500);` 强制类型安全

### 🆕 架构#1 · LoginSessionService Redis 故障期 token 残活

**文件**: `backend/src/main/java/com/smartenergy/backend/service/impl/LoginSessionService.java`  
**行号**: 46-48  
**严重度**: 🔴 CRITICAL

**当前问题**: `validateAndRefresh()` 在 Redis 故障(连不上)时返回 `true` 放行(fail-open),但 `claim()` 在 Redis 故障时抛异常 → **Redis 故障期间已登出 token 仍可继续使用**(因为撤销名单读不出来)。

**失败场景**:
1. 用户 A 登出,系统试图 `loginSession.invalidate(token)` 但 Redis 挂了 → 异常被吞或重试失败
2. Redis 恢复后,A 的 token 仍在客户端
3. A 重新请求 → `validateAndRefresh` 现在能查到撤销名单,但中间窗口 Redis 故障期所有"未刷新"的 token 都继续有效

**修复方向**: `validateAndRefresh` 改为 fail-closed:Redis 故障 → 返回 false,登录会话失效,强制重新登录。
