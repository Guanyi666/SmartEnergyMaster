## 一、项目背景与目标

### 1.1 项目现状

智驭能效（SmartEnergyMaster）当前已完成 Sprint 1 基础架构搭建，实现了以下核心能力：

- **后端**：Spring Boot 3.4.5 + MyBatis-Plus + Spring Security + JWT，22+ REST API 端点
- **前端**：Vue 3 + Element Plus + ECharts，7 个页面（登录/大屏/分析/设备管理/维修中心/人员管理/智能调度）
- **数据**：TimescaleDB 时序数据库 + Redis 缓存/限流/分布式锁
- **仿真**：Python 设备仿真器（3 设备耦合物理模型 + 故障注入 + 24h 历史回填）
- **AI**：LSTM 能耗预测服务（FastAPI，端口 8000）+ MC Dropout 不确定性估计
- **部署**：Docker Compose（TimescaleDB + Redis + 后端）

Sprint 2 规划了 LLM 多 Agent 故障诊断、动态基线异常检测、NLG 智能报告生成等高级功能。

### 1.2 建设目标

在现有系统之上构建一个 **系统级智能体（System AI Agent）**，作为整个平台的统一智能中枢，使其能够：

1. **理解**：理解操作员的自然语言指令和系统运行状态
2. **分析**：结合实时数据、历史趋势、知识库进行多维度分析
3. **推理**：基于故障模式、设备特性、业务规则进行根因推理
4. **决策**：生成操作建议、维修方案、调度策略
5. **执行**：在授权范围内自动执行操作（查询/建议/辅助执行/自主执行）

## 第一部分：可行性评估

### 二、技术可行性

#### 2.1 当前系统是否具备接入大模型的条件

**结论：✅ 已充分具备。**

| 条件 | 当前状态 | 评价 |
|------|---------|------|
| 标准化 API | 22+ REST 端点，全部有 Swagger 注解 | ✅ 可直接注册为 Agent Tool |
| Python 服务架构 | 已有 FastAPI 预测服务（端口 8000） | ✅ 可在同架构下扩展 Agent 服务 |
| 数据可访问性 | TimescaleDB + Redis，MyBatis-Plus 数据访问层 | ✅ Agent 可通过 Service 层或 API 层访问 |
| 认证体系 | JWT + Spring Security，RBAC（ADMIN/OPERATOR/MAINTENANCE_ENGINEER） | ✅ Agent 可继承用户权限 |
| 实时数据流 | 仿真器 3s 推送 + 大屏 5s 轮询 | ✅ 满足近实时 Agent 监控需求 |

**接入 LLM 的工作量评估**：仅需新增一个 Agent Orchestrator 模块（Spring Boot Service 层或独立 Python FastAPI 服务），将现有的 Service/Controller 方法封装为 LLM 可调用的 Tool 函数。核心工作量在 Tool 定义和 Prompt 工程，不涉及现有系统重构。

#### 2.2 数据是否足够支撑智能决策

**结论：⚠️ 部分具备，需分阶段完善。**

| 数据类型 | 当前状态 | 能否支撑 Agent | 补充计划 |
|---------|---------|---------------|---------|
| 实时传感器数据 | ✅ 3 设备 × 3s 频率 | 能支撑实时监控 Agent | — |
| 历史时序数据 | ✅ 24h × 96 点 × 3 设备 | 能支撑短期趋势分析 | 需积累 ≥30 天数据用于基线建模 |
| 故障案例标注 | ❌ 无标注数据 | 不能支撑监督学习 | Phase 1 用规则引擎 + LLM 推理；逐步积累 |
| 设备手册/SOP | ❌ 未数字化 | 不能支撑 RAG | Phase 1 构建知识库 |
| 电价政策 | ✅ 仿真器内置分时电价映射 | 能支撑电价相关建议 | 接入真实电网电价 API |
| 维修记录 | ✅ 工单表含故障类型/处理人/时间 | 能支撑工单分析 | 需补充维修结果和根因字段 |

**数据策略**：Phase 1 依赖规则引擎 + LLM 通用推理能力（零样本/少样本）；Phase 2-3 逐步积累标注数据，过渡到数据驱动的监督/强化学习。

#### 2.3 是否能够实现 Tool Calling

**结论：✅ 完全可以，且天然适配。**

现有系统的 22+ REST API 端点天然满足 Tool Calling 的接口规范：

```
Tool = {
  "name": "query_device_list",
  "description": "查询所有设备列表，获取实时状态、最新传感器数据和活跃工单数",
  "parameters": {
    "page": "页码（默认1）",
    "size": "每页条数（默认20）",
    "type": "设备类型过滤（ARC_FURNACE/PUMP/COMPRESSOR）",
    "status": "状态过滤（RUNNING/STOPPED/FAULT/OFFLINE）",
    "keyword": "设备名称/编码搜索"
  },
  "endpoint": "GET /api/devices"
}
```

主流 Agent 框架（LangChain、LlamaIndex、OpenAI Function Calling）均支持将 REST API 注册为 Tool。22 个端点的 Tool 定义工作量约 0.5 人天。

#### 2.4 是否适合采用 Agent 架构

**结论：✅ 非常适合。理由如下：**

1. **天然多模块结构**：系统具有设备管理、传感器采集、故障检测、工单管理、仪表盘、调度建议、预测服务 7 个功能域，每个域可对应一个专业子 Agent
2. **现有 Dashboard 聚合模式可复用**：DashboardController 聚合各 Service 的模式，与 Agent Orchestrator 聚合子 Agent 的模式高度一致
3. **人机协同已存在**：工单状态变更、调度建议确认等操作已有"人工确认"环节，Agent 可在此基础上增加 AI 建议层
4. **业务流程序列化**：故障检测→工单创建→人员指派→维修执行→闭环验证 是天然的工作流（Workflow），适合 Agent 编排

#### 2.5 是否需要构建知识库（RAG）

**结论：✅ 需要。建议构建以下 4 个知识库：**

| 知识库 | 内容 | 数据来源 | 构建方式 | 优先级 |
|--------|------|---------|---------|--------|
| 设备手册库 | 电弧炉/水泵/空压机的技术参数、正常运行工况范围、维护周期 | 设备厂商文档、运维手册 | 文档切片 + Embedding | P1 |
| 故障案例库 | 历史故障-根因-解决方案三元组，含传感器数据快照 | 工单记录 + 人工标注 | 结构化存储 + 向量检索 | P1 |
| SOP 规程库 | 标准操作流程（启停、巡检、维修、应急处理） | 企业 SOP 文档 | 文档切片 + Embedding | P2 |
| 电价政策库 | 分时电价时段表、尖峰电价触发条件、政府补贴政策 | 电网公司公告 | 结构化存储 | P2 |

#### 2.6 是否需要 MCP（Model Context Protocol）

**结论：✅ 推荐引入。MCP 作为 Agent 与业务系统的标准化接口层。**

MCP 协议将系统能力封装为三种原语：
- **Tools**：Agent 可调用的操作（对应 REST API 的 POST/PUT/PATCH/DELETE）
- **Resources**：Agent 可读取的数据（对应 REST API 的 GET 请求）
- **Prompts**：预定义的提示模板（对应特定的分析/报告生成场景）

结合本项目的 22+ 端点，MCP Server 可作为一个独立的 Spring Boot Starter 模块（`smartenergy-mcp`）实现，无需引入第三方 MCP 框架。

#### 2.7 是否需要工作流引擎（可待后续完善）

**结论：✅ 需要。复杂业务流程需要确定性编排。**

以下场景天然适合工作流：
- **故障处理工作流**：传感器异常→故障检测→工单创建→人员匹配→指派→维修→闭环
- **日报生成工作流**：数据采集→指标计算→图表生成→文本生成→报告组装
- **预测性维护工作流**：数据采集→趋势分析→健康度计算→维护建议生成

推荐使用 **LangGraph**（Python）或自研轻量 Workflow DSL（Java），无需引入重型 BPMN 引擎。

#### 2.8 是否需要多智能体架构

**结论：⚠️ 建议分阶段。Phase 1-2 单体 Agent 足够，Phase 3-4 引入多 Agent。**

- **Phase 1-2（单体 Agent）**：单一 Agent + RAG + Tool Calling，覆盖 80% 的查询和分析场景
- **Phase 3-4（多 Agent）**：随功能复杂度增长，引入 Orchestrator + 子 Agent 集群，每个子领域独立优化

### 三、业务可行性

#### 3.1 业务场景适配矩阵

| 业务场景 | AI 接管程度 | 说明 |
|---------|-----------|------|
| 数据查询与报表生成 | 🤖 AI 自主执行 | 查询设备状态、传感器数据、工单列表、生成日报，100% 可自动化 |
| 定时巡检与异常初筛 | 🤖 AI 自主执行 | 基于阈值+动态基线实时扫描，发现异常自动告警 |
| 工单自动分派（技能匹配） | 🤖 AI 辅助执行 | AI 推荐最佳人选，操作员一键确认 |
| 运行指标阈值监控 | 🤖 AI 自主执行 | 超过阈值自动告警+创建工单（现有逻辑） |
| 能耗趋势分析与预测 | 🤖 AI 自主输出 | LSTM 模型预测 + AI 解读预测结果 |
| 设备控制参数调整 | 🧑 人工决策 + AI 建议 | AI 建议目标值，操作员在 SCADA 中手动执行 |
| 高风险维修方案制定 | 🧑 人工决策 + AI 建议 | AI 提供案例参考和方案对比，人工最终拍板 |
| 生产排程与负荷转移 | 🧑 人工决策 + AI 建议 | 涉及安全和经济损失，AI 仅做模拟推演 |
| 紧急停机决策 | 🧑 纯人工决策 | 涉及人身安全，AI 禁止参与决策 |
| 设备物理参数修改 | 🧑 纯人工决策 | 如电弧炉功率上限，AI 禁止建议 |

#### 3.2 AI 接管程度分级定义

| 级别 | 名称 | 定义 | 示例 |
|------|------|------|------|
| L0 | 纯人工 | AI 不参与 | 紧急停机 |
| L1 | AI 建议 | AI 输出建议，人工决定是否采纳 | 维修方案推荐 |
| L2 | AI 辅助执行 | AI 准备参数+建议，人工一键确认 | 工单指派、调度建议采纳 |
| L3 | AI 自主执行 | AI 在授权范围内自动执行 | 数据查询、报表生成、阈值告警 |

### 四、安全与风险分析

#### 4.1 风险矩阵

| 风险类型 | 风险描述 | 发生概率 | 影响程度 | 风险等级 | 应对措施 |
|---------|---------|---------|---------|---------|---------|
| AI 误操作 | Agent 错误地创建/关闭工单或修改设备状态 | 中 | 高 | 🔴 高 | L2+操作默认需人工确认；建立操作审计日志；提供一键回滚 |
| 模型幻觉 | LLM 生成错误的故障诊断结论或虚构的数据 | 高 | 中 | 🟠 中 | 关键数值型输出要求引用数据来源；标注置信度；不确定时建议人工介入 |
| 数据泄露 | LLM 调用时传输敏感数据（设备参数、生产数据） | 中 | 高 | 🔴 高 | 数据脱敏后传输；使用本地部署模型；API 调用走内网 |
| 权限越界 | Agent 以高权限执行了用户本无权执行的操作 | 低 | 高 | 🟠 中 | Agent 继承当前用户 RBAC 权限；所有 Tool Call 携带用户 JWT Token |
| 过度依赖 | 操作员过度信任 AI 建议，放弃独立判断 | 中 | 中 | 🟡 低 | AI 输出明确标注"AI 建议，仅供参考"；定期培训操作员保持判断力 |
| 系统崩溃 | Agent 服务故障导致正常操作流程中断 | 低 | 高 | 🟠 中 | Agent 作为旁路服务，不替换原有操作界面；故障时自动降级到原 UI |

#### 4.2 应急回滚机制

- **操作审计日志**：Agent 执行的每次写操作记录到 `agent_audit_log` 表（操作人、时间、内容、Agent 版本、Token 消耗）
- **一键回滚**：提供 `/api/agent/undo/{auditId}` 接口，撤销 Agent 最近一次写操作
- **降级开关**：提供 `/api/agent/disable` 管理接口，紧急情况下可一键禁用 Agent 功能，系统回退到纯人工操作模式
- **人工接管**：Agent 在执行链路的任意节点都可以被操作员中断并接管

#### 4.3 数据安全设计

```
用户操作 → Agent 接收指令
              │
              ├─ 数据查询：通过 API 获取原始数据
              │      │
              │      └─ 发送给 LLM 前进行脱敏处理
              │         （设备编码脱敏、去除精确坐标、聚合数值）
              │
              └─ 数据写入：Agent 以用户身份执行
                     │
                     └─ 所有写操作记录 audit_log
                        含：user_id, agent_version, tool_name, input_params, output, timestamp
```

### 六、生产监控智能体（Production Monitor Agent）

> **接管程度**：AI 自主执行（L3）
> **核心职责**：7×24 实时监控设备运行状态，发现异常主动告警，生成运行报告

#### 6.1 二级能力：实时状态监控

| 编号 | 具体操作 | 调用工具 | AI 能力 | 输出形式 |
|------|---------|---------|--------|---------|
| PM-01 | 查询全厂设备实时运行状态（运行/停机/故障/离线数量） | T-001（设备列表） | 汇总统计 + 状态解读 | 自然语言摘要 + 状态卡片 |
| PM-02 | 查询指定设备的最新传感器数据（温度/振动/压力/功耗） | T-003（最新传感器） | 数据解读 + 异常标注 | 仪表盘数据 + 文字说明 |
| PM-03 | 查询全厂总有功功率、累计碳排放、当前电价区间 | T-005（仪表盘总览） | 指标解读 + 对比昨日同期 | 指标卡片 + 趋势描述 |
| PM-04 | 大屏数据异常主动推送告警 | T-003 + 内部规则引擎 | 对比历史基线，发现异常偏离 | 主动告警消息 |

**示例对话**：
> 用户："现在全厂运行情况怎么样？"
> Agent："当前全厂 3 台设备中，EAF-01 正在运行（负荷 85%），PUMP-01 和 COMP-01 处于停机状态。总有功功率 1,250 kWh，当前处于峰时电价（1.2 元/kWh）。EAF-01 温度 850°C，在正常范围内。有一个待处理工单：EAF-01 机械卡涩告警（HIGH 优先级）。"

#### 6.2 二级能力：趋势分析与预警

| 编号 | 具体操作 | 调用工具 | AI 能力 | 输出形式 |
|------|---------|---------|--------|---------|
| PM-05 | 查询设备历史时序数据并分析趋势 | T-004（历史传感器） | 趋势分析 + 周期性识别 | 趋势图 + 分析文字 |
| PM-06 | 对比当前读数与历史同期，发现异常偏离 | T-004 + 内部统计引擎 | 同期对比 + 偏差计算 | 异常标注 + 偏差量 |
| PM-07 | 基于 LSTM 预测未来 30 分钟负荷趋势 | T-007（能耗预测） | 预测解读 + 置信区间说明 | 预测曲线 + 文字解读 |
| PM-08 | 电价时段切换提醒 | T-005（电价区间） | 时段变更主动通知 | 提前 15 分钟推送提醒 |

### 七、故障诊断智能体（Fault Diagnosis Agent）

> **接管程度**：AI 建议 + AI 辅助执行（L1-L2）
> **核心职责**：多维度传感器交叉验证，根因推理，生成结构化维修建议

#### 7.1 二级能力：自动故障检测

| 编号 | 具体操作 | 调用工具 | AI 能力 | 输出形式 |
|------|---------|---------|--------|---------|
| FD-01 | 实时执行故障检测规则（静态阈值 + 动态基线） | T-003 + 内部规则引擎 | 多规则并行执行 + 告警聚合 | 告警列表 + 严重度排序 |
| FD-02 | 多维传感器交叉验证（温度/振动/压力/功耗/CO₂） | T-003 + T-004 | 交叉验证矩阵匹配 | 验证结果 + 匹配的故障模式 |
| FD-03 | 基于历史统计的动态基线异常检测（滑动窗口均值 ± 3σ） | T-004 + 统计引擎 | 动态基线计算 + 异常点标记 | 异常点列表 + 偏离程度 |
| FD-04 | 计算设备健康度评分（0-100）并跟踪衰退趋势 | T-014（健康度评分） | 评分解读 + 趋势预测 | 健康度仪表 + 文字说明 |

**故障交叉验证矩阵**（现有 2 条 → Agent 扩展到 10 条）：

| 故障模式 | 温度 | 振动 | 压力 | 功耗 | 运行状态 | 根因 | 紧急度 |
|---------|------|------|------|------|---------|------|--------|

```
   "检测到冷却中断（置信度 92%），已创建工单 WO-2024-0042，
    推荐指派张三。是否确认？"
```

#### 13.2 Agent 间通信模式

| 模式 | 场景 | 说明 |
|------|------|------|
| Task Delegate（任务委托） | Orchestrator → 子 Agent | 主控将子任务分配给专业 Agent，等待返回 |
| Result Callback（结果回调） | 子 Agent → Orchestrator | 子 Agent 完成任务后返回结构化结果 |
| Context Share（上下文共享） | 所有 Agent ↔ Memory Manager | 通过共享记忆管理器传递对话上下文 |
| Human Interrupt（人工中断） | 任意节点 → 用户 | Agent 在执行链路中可暂停等待人工确认 |

### 十五、权限分级模型

| 级别 | 名称 | 定义 | 适用 Tool | Agent 行为 |
|------|------|------|-----------|-----------|
| L0 | 自主查询 | AI 可自主执行，无需人工确认 | T-001 ~ T-011, T-014 ~ T-019, T-021, T-022 | 直接调用 API，返回结果 |
| L1 | AI 建议 | AI 生成建议方案，人工决定是否执行 | （不直接调用 Tool，仅输出建议文本） | 输出建议 + 置信度 + 数据来源 |
| L2 | 辅助执行 | AI 准备操作参数并展示，人工一键确认后执行 | T-012, T-013, T-020 | 弹出确认卡片，人工点击"确认"后执行 |
| L3 | 禁止 | AI 不得建议或执行 | 设备删除、物理控制、系统配置变更 | 拒绝执行并提示原因 |
