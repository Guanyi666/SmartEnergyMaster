## 1. 重构背景与目标

### 1.1 当前问题总结

| 问题类型 | 数量 | 典型表现 |
|---------|------|---------|
| **文件重复** | 1 对 | `01_init.sql` 与 `init.sql` 内容高度重复，`init.sql` 执行会抛异常 |
| **废弃表** | 1 张 | `maintenance_personnel`（来自 `02_user_expansion.sql`）未被 Java 代码使用 |
| **表名不统一** | 1 张 | `workorder_assignment` 带 `workorder_` 前缀，其他表不带 |
| **用户表字段割裂** | 前后矛盾 | `02_user_expansion.sql` 给 `sys_user` 加了 8 个字段，但 Java `SysUser` 实体只有 4 字段（id/username/password/role）——扩展字段形同虚设 |
| **人员与用户无关联** | 缺外键 | E001 在 `sys_user` 和 `workorder_maintenance_personnel` 中各有一行，但两表之间没有 FK 约束，数据一致性靠人工维护 |
| **角色体系薄弱** | 仅有 4 种角色 | 缺少设备管理员、人事等关键岗位；后端无 `@PreAuthorize` 方法级权限 |
| **设备类型英文** | 6 种类型 | `ARC_FURNACE` 等英文常量在 DB/Java/前端三处各维护一套映射，容易出现不一致 |

### 1.2 重构目标

1. **清理冗余**：删除重复文件、废弃表，统一命名规范
2. **角色升级**：从 4 角色扩展为 5 角色（维修工程师 / 设备管理员 / 操作员 / 生产经理 / 人事部）
3. **关联用户与人员**：`workorder_maintenance_personnel` 通过 `user_id` FK 关联 `sys_user`，建立一人一账号的约束
4. **补全 Java 实体**：`SysUser.java` 补全 `02_user_expansion.sql` 已有的字段，结束"DB 有列但 Java 读不到"的怪象
5. **设备类型中文化**：DB 存储中文值，Java 通过枚举统一管理映射
6. **预留扩展字段**：为迭代 2 及未来需求预埋字段

## 5. 用户表与维修人员表关联方案

### 5.1 两张表现状对比

| 字段 | `sys_user`（含 02 扩展） | `workorder_maintenance_personnel` | 关系 |
|------|:---:|:---:|:---:|
| `id` | SERIAL PK | BIGSERIAL PK | 不同表，独立主键 |
| `username` / `employee_no` | username | employee_no | 语义相同但无 FK |
| `password` | ✅ BCrypt | ❌ | 仅用户表有 |
| `nickname` / `name` | nickname | name | 语义相似 |
| `phone` | ✅ | ✅ | 可能不同（工作号 vs 私人号） |
| `email` | ✅ | ✅ | 可能不同（公司邮箱 vs 个人邮箱） |
| `role` | ✅ | ❌ | 仅用户表有 |
| `specializations` | ❌ | ✅ JSONB | 仅人员表有 |
| `skill_level` | ❌ | ✅ | 仅人员表有 |
| `certification` | ❌ | ✅ | 仅人员表有 |
| `current_workload` | ❌ | ✅ | 仅人员表有 |
| `max_workload` | ❌ | ✅ | 仅人员表有 |
| `is_on_duty` | ❌ | ✅ | 仅人员表有 |
| `avatar_color` | ❌ | ✅ | 仅人员表有 |
| `avatar_url` | ✅ | ❌ | 仅用户表有 |
| `status` | ✅ | ❌ | 仅用户表有 |
| `last_login_at` | ✅ | ❌ | 仅用户表有 |

### 5.2 为什么不合并

经过代码审查，**不合并**的理由如下：

**（1）改动成本过高**

`MaintenancePersonnel` 实体被 12 个后端文件引用——包括指派服务（`WorkOrderAssignmentServiceImpl` 使用了 30+ 次 `personnelMapper`）、自动匹配引擎（`AutoMatchEngine.scorePersonnel(p)` 以 `MaintenancePersonnel` 为入参）、调度看板统计、Controller CRUD 等。合并意味着这些全部重写。

| 方案 | 后端改动文件 | 前端改动文件 | 风险 |
|------|:---:|:---:|:---:|
| 合并 | ~14 个 | ~5 个 | 高（指派/匹配/负载逻辑全改） |
| **关联（本文案）** | **3 个** | **2 个** | **低（现有逻辑不动）** |

**（2）业务语义不同**

`sys_user` = 认证与授权（这个人能登录吗？有什么权限？）
`workorder_maintenance_personnel` = 专业技能档案（这个人会修什么？当前忙不忙？）

两表职责不同。合并会导致 `specializations`、`current_workload` 等 6 个字段对 80% 的非维修用户永远是 NULL——这是典型的"稀疏列"坏味道。

**（3）企业系统的既有模式**

SAP 的 HR 模块（PA）与用户管理（SU01）分表，Oracle EBS 的 `PER_ALL_PEOPLE_F` 与 `FND_USER` 分表。工业系统的惯例是"人员主数据"和"登录账号"分离——因为不是每个员工都需要登录系统，也不是每个系统账号都对应一个自然人。

**（4）当前代码已在正常运行**

指派、匹配、负载跟踪的整套逻辑工作正常。合并带来的是重写风险，而非功能增益。

### 5.3 实际方案：关联 + 去重

**核心思路**：保持两表独立，做三件事：

1. **`workorder_maintenance_personnel` 新增 `user_id` FK → `sys_user.id`**：建立"一个维修人员 = 一个 sys_user 账号"的约束
2. **`SysUser.java` 实体补全字段**：把 `02_user_expansion.sql` 已经在 DB 中存在的 nickname/phone/email/status/department/position 等字段加入 Java 实体
3. **种子数据保持 E001 双表模式**：`sys_user` 中有 E001 登录账号，`workorder_maintenance_personnel` 中有 E001 技能档案，`user_id` 关联二者

```
sys_user                              workorder_maintenance_personnel
├── id (PK) = 2                       ├── id (PK) = 1
├── username = 'E001'  ──逻辑一致──▶  ├── employee_no = 'E001'
├── password (BCrypt)                 ├── user_id (FK) = 2  ──FK──▶ sys_user.id
├── role = 'MAINTENANCE_ENGINEER'     ├── name = '张工'
├── nickname = '张工'                 ├── phone
├── phone                            ├── email
├── email                            ├── specializations (JSONB)
├── department 🆕                    ├── skill_level
├── position 🆕                      ├── certification
├── hire_date 🆕                     ├── current_workload
├── status                           ├── max_workload
├── is_deleted 🆕                    ├── is_on_duty
├── last_login_at                    ├── avatar_color
├── created_at                       ├── created_at
└── updated_at                       └── updated_at
```

**关键约束**：
- `workorder_maintenance_personnel.user_id` 可为 NULL（兼容历史数据或临时人员记录）
- 当 `user_id` 有值时，`employee_no` 必须等于对应 `sys_user.username`（应用层校验，避免 DB 触发器复杂度）
- `workorder_assignment.personnel_id` **不动**——继续指向 `workorder_maintenance_personnel.id`，因为指派的对象是"维修人员"这个专业角色，不是泛化的"用户"

### 5.4 这样改的好处

| 好处 | 说明 |
|------|------|
| **改动最小** | 后端仅改 SysUser 实体（加字段）+ personnel 表（加一个 FK），现有指派/匹配/负载逻辑一行不改 |
| **数据一致** | `user_id` FK 确保每个 personnel 对应一个有效 sys_user，不会出现"幽灵人员" |
| **一人一账号** | 维修人员通过 `username = employee_no` 登录，无需额外映射 |
| **职责清晰** | 人事部管 `sys_user`（账号启停），设备运维主管管 `workorder_maintenance_personnel`（技能/负载） |
| **可扩展** | 未来若有"外部维修承包商"不需要登录系统，只需 `user_id = NULL` 即可 |

## 8. 迁移方案

### 8.1 全新部署（推荐）

```bash
cd deploy
docker-compose down -v          # 彻底清空数据卷
# 替换 01_init.sql 为本文档 DDL
# 新增 02_seed_data.sql
# 删除 init.sql, seed_cases.sql, seed_sop.sql
docker-compose up -d            # 重新初始化
```

### 8.2 从 V1 迁移（保留数据）

```sql
-- Step 1: sys_user 补全字段（02_user_expansion.sql 中已有大部分，此处确保列存在）
ALTER TABLE sys_user ADD COLUMN IF NOT EXISTS department VARCHAR(64);
ALTER TABLE sys_user ADD COLUMN IF NOT EXISTS position VARCHAR(64);
ALTER TABLE sys_user ADD COLUMN IF NOT EXISTS hire_date DATE;
ALTER TABLE sys_user ADD COLUMN IF NOT EXISTS is_deleted BOOLEAN DEFAULT FALSE;

-- Step 2: workorder_maintenance_personnel 新增 user_id FK
ALTER TABLE workorder_maintenance_personnel
    ADD COLUMN IF NOT EXISTS user_id INTEGER REFERENCES sys_user(id) ON DELETE SET NULL;

-- 回填 user_id：根据 employee_no 匹配 sys_user.username
UPDATE workorder_maintenance_personnel p
    SET user_id = u.id
    FROM sys_user u
    WHERE p.employee_no = u.username AND p.user_id IS NULL;

-- Step 3: 更新 device_type 为中文
UPDATE device          SET device_type = '电弧炉'    WHERE device_type = 'ARC_FURNACE';
UPDATE device          SET device_type = '循环水泵'  WHERE device_type = 'PUMP';
UPDATE device          SET device_type = '空压机'    WHERE device_type = 'COMPRESSOR';
UPDATE device          SET device_type = '钢包精炼炉' WHERE device_type = 'LADLE_FURNACE';
UPDATE device          SET device_type = '连铸机'    WHERE device_type = 'CONTINUOUS_CASTER';
UPDATE device          SET device_type = '除尘系统'  WHERE device_type = 'DUST_COLLECTOR';
UPDATE maintenance_sop SET device_type = '电弧炉'    WHERE device_type = 'ARC_FURNACE';
UPDATE maintenance_sop SET device_type = '循环水泵'  WHERE device_type = 'PUMP';
UPDATE maintenance_sop SET device_type = '空压机'    WHERE device_type = 'COMPRESSOR';
UPDATE repair_case     SET device_type = '电弧炉'    WHERE device_type = 'ARC_FURNACE';
UPDATE repair_case     SET device_type = '循环水泵'  WHERE device_type = 'PUMP';
UPDATE repair_case     SET device_type = '空压机'    WHERE device_type = 'COMPRESSOR';

-- Step 4: 更新 sys_user 角色值
UPDATE sys_user SET role = 'EQUIPMENT_ADMIN'    WHERE role = 'ADMIN';
UPDATE sys_user SET role = 'PRODUCTION_MANAGER' WHERE role = 'MANAGER';

-- Step 5: 重命名 workorder_assignment → work_order_assignment（幂等）
DO $$ BEGIN
    IF EXISTS (SELECT 1 FROM information_schema.tables
               WHERE table_name = 'workorder_assignment') THEN
        ALTER TABLE workorder_assignment RENAME TO work_order_assignment;
    END IF;
END $$;

-- Step 6: 清理废弃表
DROP TABLE IF EXISTS maintenance_personnel CASCADE;
```

### 8.3 注意事项

1. **Breaking Change**：角色值（ADMIN→EQUIPMENT_ADMIN+HR）和设备类型（英文→中文）变更**无法向后兼容**，后端和前端必须同步更新
2. **现有逻辑保护**：`workorder_maintenance_personnel` 表结构不变（仅加一列），12 个依赖它的后端文件无需修改
3. **回滚策略**：迁移前执行 `pg_dump` 全量备份
4. **测试验证**：迁移后重点验证维修人员 CRUD、指派/释放工单、自动匹配引擎功能正常
