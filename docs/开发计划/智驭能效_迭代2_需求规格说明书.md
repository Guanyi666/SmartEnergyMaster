### 2.2 现存问题与差距

经过对迭代1交付物的全面审查，识别出以下核心问题：

#### 2.2.1 功能孤岛问题（严重）

当前工单管理、SOP 知识库、维修案例库、备件库存四大模块**各自独立运行**，缺乏业务闭环：

| 问题 | 现状 | 影响 |
|------|------|------|
| 工单→SOP 断裂 | 查看工单时不会自动推荐匹配的 SOP，需手动去知识库搜索 | 维修效率低，新手不知如何处理 |
| 工单→案例断裂 | 工单详情页不展示相似历史案例，无法借鉴前人经验 | 重复踩坑，知识未沉淀 |
| 工单→备件断裂 | 维修过程中领用备件与工单无直接关联，需切换页面操作 | 操作繁琐，事后追溯困难 |
| SOP→备件断裂 | SOP 定义了 `required_parts` 但无自动校验库存是否充足 | 按 SOP 操作时可能发现缺件 |
| 案例→SOP 断裂 | 案例闭环后不会反向优化 SOP（如更新步骤、延长预估时间） | SOP 持续老化，未形成知识循环 |

#### 2.2.2 前端交互体验问题（严重）

| 问题 | 具体表现 |
|------|---------|
| **导航层级混乱** | 10 个页面平铺在侧边栏，无分组逻辑，操作员/管理员/维修工程师看到相同菜单（仅"维修指挥中心"和"人员调度"对 OPERATOR 隐藏） |
| **信息密度失衡** | Dashboard 大屏信息丰富但 AnalysisView 仅有基础折线图；DevicesView 同时承载设备管理+工单列表，职责不清 |
| **窄屏适配差** | 侧边栏 280px 固定宽度，在 ≤1024px 屏幕下变为顶部堆叠；备件/知识库表格在 768px 以下列重叠（已有部分修复但不彻底） |
| **视觉层级弱** | 全局使用同一套玻璃拟态风格，缺乏信息层级区分；关键告警与普通信息视觉权重相同 |
| **操作路径长** | 从"发现告警"到"查看工单→匹配SOP→领用备件→记录案例"需在 4 个页面间跳转 |

#### 2.2.3 角色权限问题（严重）

| 问题 | 现状 |
|------|------|
| **后端无 RBAC** | Spring Security 仅区分"已登录/未登录"，所有已认证用户可调用全部 API——操作员理论上可删除设备、修改 SOP |
| **前端仅有粗粒度守卫** | `router.beforeEach` 仅对 `/maintenance` 等 3 个路由做了角色限制，视图内部无差异化渲染 |
| **SCHEDULER 角色缺失** | 文档定义了 5 种角色，但代码中仅实现 4 种（ADMIN/OPERATOR/MAINTENANCE_ENGINEER/MANAGER），调度员未落地 |
| **5 角色无专属首页** | 所有非维修角色统一落地 `/dashboard`，管理层、调度员、操作员看到完全相同的大屏内容 |
| **菜单一刀切** | 操作员能看到"维修知识库"、"备件库存"菜单并执行 CRUD，这超出了操作员的职责范围 |

#### 2.2.4 缺失的数字孪生可视化

当前 Dashboard 大屏以指标卡片 + 仪表盘图表 + 设备状态磁贴呈现，缺乏空间感知——用户无法直观理解设备在工厂中的物理布局、物流走向（废钢→电弧炉→精炼炉→连铸机）、能源流（电→电弧炉/水泵/空压机）和冷却水流向。

#### 2.2.5 其他技术债务

| 问题 | 影响 |
|------|------|
| 无单元测试 | 仅有 `contextLoads()` 占位测试，业务逻辑无回归保护 |
| 审计日志表存在但无 UI | `audit_log` 表已建但无查询/展示页面 |
| 通知表存在但无推送 | `notification` 表已建但告警仅在前端轮询，无主动推送机制 |
| 碳配额表存在但无管理界面 | `carbon_quota` 表已建但无可视化配置页面 |
| 无操作确认/撤销机制 | 删除设备、删除 SOP 等破坏性操作无二次确认（前端部分有，后端无软删除） |

### 2.3 用户反馈汇总

基于迭代1评审与干系人访谈，核心诉求如下：

> **"我是维修工程师，接到工单后希望在一个页面完成全部操作——看故障现象、查 SOP、找相似案例、申领备件——而不是在四个页面之间反复横跳。"**  
> — 维修工程师角色诉求

> **"我想一眼看到整个工厂的样子——电弧炉在哪儿、水泵在哪儿、哪台设备红了、物料从哪里流到哪里——而不是盯着枯燥的数字表格。"**  
> — 管理层角色诉求

> **"现在菜单太乱了，操作员不应该看到维修知识库的编辑按钮。不同的人进来应该看到不同的东西。"**  
> — 系统管理员角色诉求

> **"屏幕小一点的时候，表格就挤成一团，根本没法用。现场工控机很多是 1366×768 的老屏幕。"**  
> — 操作员角色诉求

## 3. 迭代2 总体目标与策略

### 3.1 迭代主题

**迭代主题**：智能联动 · 数字孪生 · 多角色体验

**一句话目标**：以工单为纽带打通 SOP-案例-备件全链路，构建数字孪生工厂可视化，按五角色重构前端体验，将系统从"功能可用"升级为"业务好用"。

### 3.2 核心设计原则

| 原则 | 说明 |
|------|------|
| **一页到底（One-Page Fulfillment）** | 围绕工单处理的核心场景，用户在一个页面内完成"诊断→查SOP→参考案例→申领备件→记录结果"全流程 |
| **空间即信息（Spatial Dashboard）** | 用 2.5D 等轴测图呈现工厂物理拓扑，设备状态以色温编码直接映射到视觉，替代纯数字罗列 |
| **角色即视图（Role = View）** | 不同角色登录后看到不同的首页、菜单、数据范围、操作权限——而非同一套界面做减法 |
| **渐进式复杂度（Progressive Disclosure）** | 默认展示最重要的 3-5 条信息，详情通过展开/弹窗/抽屉渐进呈现 |
| **现场优先（Field-Ready）** | 适配 1366×768 工控机屏幕，触控友好，关键操作按钮不小于 44×44px |

## 4. 需求详述

### 4.1 需求一：工单-SOP-案例-备件一体化闭环

#### 4.1.1 需求概述

打破当前四大模块各自独立的局面，以**工单为业务枢纽**，实现以下联动：

```
工单详情页（维修工作台）
├── ① 自动匹配 SOP     → 展示 Top 3 匹配 SOP，可展开查看步骤
├── ② 相似案例推荐     → 展示 Top 5 相似历史案例，含维修结果
├── ③ 备件快捷申领     → 根据 SOP 的 required_parts 一键申领，自动扣减库存
├── ④ AI 维修建议       → 综合 SOP + 案例 + 实时传感器数据，LLM 生成诊断建议
└── ⑤ 闭环→生成案例    → 工单 RESOLVED 后提示"是否生成维修案例"，自动回填数据
```

#### 4.1.2 功能详述

##### F1.1 工单详情页重塑 —「维修工作台」

**现状**：工单详情以 Drawer（右侧抽屉）展示，包含设备快照、传感器指标、指派人、操作按钮，但没有 SOP/案例/备件联动。  
**改造**：将工单详情的 Drawer 升级为**全功能维修工作台**，采用 Tab 分组结构：

| Tab | 内容 | 数据来源 |
|-----|------|---------|
| **故障概览** | 工单基本信息、设备状态、故障时刻传感器快照（温度/振动/压力仪表盘）、状态时间线 | 现有 `GET /api/workorder/orders/{id}` + sensor_data |
| **SOP 指引** | 自动匹配的 Top 3 SOP，默认展开最佳匹配。每项显示：SOP 标题、版本号、预计耗时、所需技能、逐步操作说明。支持"采纳此 SOP"按钮，将 SOP 绑定到工单（更新 work_order.sop_id） | `GET /api/sops/match?deviceType=&faultType=` |
| **相似案例** | Top 5 相似历史案例卡片。每项显示：案例标题、故障现象、根因、维修结果、耗时、维修人。支持点击跳转案例详情 | `GET /api/cases/similar?deviceType=&faultType=&keywords=` |
| **备件申领** | 根据 SOP 的 `required_parts` 自动列出所需备件清单，每项显示：备件名称/规格/当前库存/需求量/库存是否充足。支持"一键申领"（批量创建 spare_part_usage 记录并关联工单）。低于安全库存的备件标红提示 | `GET /api/spare-parts` + SOP.required_parts + `POST /api/spare-parts/usage` |
| **AI 诊断** | 综合实时传感器数据 + 匹配的 SOP + 相似案例，生成三段式诊断报告：**异常现象 → 根因分析 → 处置建议**。支持"重新生成"。LLM 不可用时降级为基于规则的建议 | `POST /api/ai/repair-advice`（已有，需增强） |

**交互规范**：
- Tab 切换不销毁组件（`keep-alive`），保留用户操作状态
- 备件申领 Tab 完成后，其他 Tab 的"所需备件"库存数据实时刷新
- 采纳 SOP 后，SOP 指引 Tab 高亮"已采纳"状态

##### F1.2 备件申领与工单自动关联

**现状**：备件领用记录（`spare_part_usage`）有 `work_order_id` 外键字段，但前端领用流程与工单无关联——用户需在备件页面手动选择备件填写领用单，无法追溯到具体工单。

**改造**：
1. 工单维修工作台的备件申领 Tab：自动关联当前工单 ID
2. 备件库存页面增加"关联工单"筛选条件，可按工单号查询领用记录
3. 工单闭环时，自动检查是否有领用记录：若无，提示"本工单未申领备件，是否确认闭环？"
4. 备件库存不足时，自动触发低库存通知（插入 `notification` 表，见需求五）

**API 变更**：

| 变更类型 | 端点 | 说明 |
|---------|------|------|
| 新增 | `GET /api/work-orders/{id}/sop-matches` | 返回该工单（基于 device_type + fault_type）匹配的 SOP 列表 |
| 新增 | `GET /api/work-orders/{id}/similar-cases` | 返回该工单的相似案例列表 |
| 新增 | `GET /api/work-orders/{id}/required-parts` | 返回该工单已绑定 SOP 所需的备件清单及库存状态 |
| 新增 | `POST /api/work-orders/{id}/bind-sop` | 将指定 SOP 绑定到工单（更新 sop_id） |
| 增强 | `POST /api/spare-parts/usage` | 批量领用模式：接受数组 payload，一次请求申领多个备件 |
| 新增 | `GET /api/spare-parts/usage?workOrderId=` | 按工单查询领用记录 |
| 增强 | `POST /api/ai/repair-advice` | 增加相似案例作为 prompt 上下文，提升诊断质量 |

##### F1.3 工单闭环→自动生成案例

**流程**：
1. 操作员/维修工程师将工单状态变更为 `RESOLVED`
2. 后端检测闭环操作，判断是否满足生成案例条件（该工单有完整的故障现象+维修过程记录）
3. 前端弹出提示："该工单已闭环，是否生成维修案例以供后续参考？"
4. 用户确认后，调用 `POST /api/cases/from-work-order/{workOrderId}`（已有 API），自动填充：
   - `title`、`device_type`、`fault_type` ← 工单信息
   - `fault_symptom` ← 工单 description

### 4.2 需求二：数字孪生工厂可视化

#### 4.2.1 需求概述

构建一个 **2.5D 等轴测视角（Isometric View）** 的工厂数字孪生页面，以空间化方式展示 6 台设备在工厂中的物理布局、物流走向、能源流向和实时运行状态。每类设备展示一台代表，点击可跳转到该类设备的详情/列表。

#### 4.2.2 技术选型建议

| 方案 | 优势 | 劣势 | 推荐度 |
|------|------|------|--------|
| **Three.js + 自定义 2.5D 场景** | 渲染效果最佳，可做 3D 旋转/缩放；后续可升级为全 3D | 开发成本较高，需建模 | ⭐⭐⭐⭐⭐ |
| **CSS 3D Transform（Isometric）** | 纯前端实现，轻量级，不依赖 WebGL | 效果受限，复杂动画吃力 | ⭐⭐⭐⭐ |
| **ECharts GL** | 与现有 ECharts 技术栈一致，上手快 | 3D 能力有限，不太适合场景渲染 | ⭐⭐⭐ |
| **Unity WebGL** | 效果最佳，工业级 | 包体大，加载慢，与 Vue 集成困难 | ⭐⭐ |

**推荐方案**：**Three.js + CSS 3D Transform 混合方案**。

- Three.js 负责 2.5D 场景主体（工厂底板、设备模型、管道连线、粒子流动效果）
- CSS 3D Transform 负责设备状态标签、告警气泡等 UI 覆盖层
- 使用 `@vueuse/core` 的 `useResizeObserver` 实现画布自适应

### 4.4 需求四：五角色权限与专属视图

将系统从当前的"单视图 + 部分菜单隐藏"升级为完整的**基于角色的访问控制（RBAC）**，五类角色各有专属首页、差异化视图和明确的操作权限边界。

#### 4.4.2 角色定义

| 角色 | 编码 | 默认首页 | 核心职责 | 预置账号 |
|------|------|---------|---------|---------|
| **系统管理员** | `ADMIN` | `/admin/dashboard` | 用户管理、系统配置、审计日志、全局设置 | `admin` / `admin123` |
| **操作员** | `OPERATOR` | `/dashboard` | 7×24 值班监控、告警响应、工单状态跟踪 | 注册默认角色 |
| **维修工程师** | `MAINTENANCE_ENGINEER` | `/maintenance` | 接收工单、参考 SOP 执行维修、申领备件、闭环工单 | `E001` / `123456` |
| **调度主管** | `MANAGER` | `/digital-twin` | 全厂态势总览、KPI 看板、节能决策、碳排放跟踪 | 需管理员创建 |
| **调度员** | `SCHEDULER` | `/scheduler` | 生产排程优化、电价窗口分析、负荷转移建议 | 需管理员创建 |

#### 4.4.3 功能详述

##### F4.1 后端 RBAC 实现

**现状**：Spring Security 仅区分"已登录/未登录"，无方法级权限控制。  
**改造**：

**（1）角色常量定义**
```java
// 新建 com.smartenergy.backend.constant.Role.java
public final class Role {
    public static final String ADMIN = "ADMIN";
    public static final String OPERATOR = "OPERATOR";
    public static final String MAINTENANCE_ENGINEER = "MAINTENANCE_ENGINEER";
    public static final String MANAGER = "MANAGER";
    public static final String SCHEDULER = "SCHEDULER";
}
```

**（2）SecurityConfig 增加方法级安全**
```java
@EnableMethodSecurity(prePostEnabled = true)  // 启用 @PreAuthorize
```

**（3）各 Controller 方法添加权限注解**

| 操作 | 允许角色 |
|------|---------|
| 设备 CRUD（创建/编辑/删除） | `ADMIN` |
| 设备查询（列表/详情） | 全部已认证用户 |
| 工单状态变更 | `ADMIN`, `MAINTENANCE_ENGINEER`, `OPERATOR` |
| 工单创建（手动） | `ADMIN`, `MAINTENANCE_ENGINEER`, `OPERATOR` |
| 工单指派/释放/替换 | `ADMIN`, `MAINTENANCE_ENGINEER` |
| SOP CRUD | `ADMIN`, `MAINTENANCE_ENGINEER` |
| SOP 查询 | 全部已认证用户 |
| 案例 CRUD | `ADMIN`, `MAINTENANCE_ENGINEER` |
| 案例查询 | 全部已认证用户 |
| 备件 CRUD | `ADMIN` |
| 备件领用 | `ADMIN`, `MAINTENANCE_ENGINEER` |
| 备件查询 | 全部已认证用户 |
| 人员 CRUD | `ADMIN` |
| 人员查询/调度 | `ADMIN`, `MAINTENANCE_ENGINEER` |
| 用户管理 | `ADMIN` |
| 系统配置 | `ADMIN` |
| 审计日志查询 | `ADMIN` |
| 通知查询 | 全部已认证用户（各自只看自己的） |

**（4）数据权限隔离**
- `notification` 查询：自动过滤 `user_id = 当前用户ID`（除 ADMIN 可查看全部）
- `audit_log` 查询：仅 ADMIN 可访问

##### F4.2 前端路由守卫升级

**改造 `router.beforeEach`**：

```javascript
// router/index.js
const roleRouteMap = {
  ADMIN: '/admin/dashboard',
  OPERATOR: '/dashboard',
  MAINTENANCE_ENGINEER: '/maintenance',
  MANAGER: '/digital-twin',
  SCHEDULER: '/scheduler'
}

// 每个路由的 meta.roles 精确控制
// 无 meta.roles 的路由 = 所有已认证用户可访问
// meta.roles: ['ADMIN'] = 仅管理员可访问
```

### 4.5 需求五：系统增强与体验优化

以下是我（产品经理 + 技术审查）基于代码审查发现的改进点，作为迭代2的补充需求。

#### 4.5.1 通知系统落地

**现状**：`notification` 表已建但仅有 DDL，前端无通知展示、后端无推送逻辑。  
**实现**：
1. 后端：在关键事件发生时自动插入 `notification` 记录：
   - 新工单创建（通知操作员 + 维修工程师）
   - 工单被指派（通知维修工程师）
   - 工单闭环（通知创建人）
   - 备件库存低于安全库存（通知管理员 + 维修工程师）
   - 设备离线超过 5 分钟（通知操作员）
   - 调度建议生成（通知调度员）
   - 碳配额使用超 90%（通知管理员 + 管理层）
2. 前端：通知铃铛组件（已有设计，见 4.3.4），通知列表页面（`/notifications`）
3. 通知标记已读/全部已读/按类型筛选

**新增 API**：
| 端点 | 说明 |
|------|------|
| `GET /api/notifications?type=&isRead=&page=&size=` | 分页查询当前用户的通知 |
| `GET /api/notifications/unread-count` | 获取未读通知数量 |
| `PATCH /api/notifications/{id}/read` | 标记单条已读 |
| `PATCH /api/notifications/read-all` | 全部标记已读 |

**新增页面**：`NotificationsView.vue`（`/notifications`）

#### 4.5.2 操作确认与撤销

**现状**：前端部分有 `ElMessageBox.confirm`，但后端无软删除/逻辑删除，删除即物理删除不可恢复。  
**实现**：
1. 所有破坏性操作（删除设备/工单/SOP/案例/备件/人员）统一增加二次确认弹窗，提示影响的关联数据
2. 关键实体增加逻辑删除标记（`is_deleted BOOLEAN DEFAULT FALSE`）：`device`、`work_order`、`maintenance_sop`、`repair_case`、`spare_part`
3. 逻辑删除的数据在默认查询中排除（MyBatis-Plus `@TableLogic` 注解）
4. 管理员可在审计日志中查看已删除记录，支持恢复（`PATCH /api/{entity}/{id}/restore`）

#### 4.5.3 全局搜索

**实现**：
- 后端：`GET /api/search?keyword=` 全文搜索（跨设备/工单/SOP/案例/备件 5 张表，使用 PostgreSQL `ILIKE` 或 `tsvector`）
- 前端：全局搜索弹窗（Ctrl+K 唤起），输入关键词实时检索（防抖 300ms），结果按类型分组展示，点击跳转

#### 4.5.4 数据导出

**实现**：
- 设备列表导出 CSV
- 传感器历史数据导出 CSV（指定设备 + 时间范围）
- 工单列表导出 CSV
- 能效报告导出 PDF（MANAGER 角色专属，包含：能耗趋势图 + 碳排放统计 + 成本分析 + 节能建议）

#### 4.5.5 单元测试补全

基于 CLAUDE.md 中识别出的测试缺口：
- Service 层业务逻辑测试（JUnit 5 + Mockito）：至少覆盖故障检测引擎、工单状态流转、自动匹配引擎、备件库存扣减
- Controller 层接口测试（MockMvc / `@WebMvcTest`）：至少覆盖认证、设备 CRUD、传感器数据上传
- 集成测试（`@SpringBootTest`）：传感器上传→故障检测→工单创建的端到端流程
- 目标覆盖率：行覆盖率 ≥ 60%（迭代2阶段性目标）

#### 4.5.6 数据仿真器增强

- 命令行参数支持：`--interval`（推送间隔）、`--fault-rate`（故障注入概率）、`--history-hours`（回填时长）、`--devices`（指定模拟设备列表）
- 增加渐进劣化故障模式（轴承磨损 `--wear-mode`），振动和温度逐步上升，模拟真实设备老化
- 支持模拟设备启停（随机间隔的 STOPPED ↔ RUNNING 切换）
- 输出运行日志：每 100 次推送输出一次统计（成功数/失败数/故障触发次数）

#### 4.5.7 碳配额管理前端

**现状**：`carbon_quota` 表已建但无管理界面。  
**实现**：管理员在 `/admin/carbon` 页面管理年度/月度碳配额，查看配额使用进度条（实际排放 / 配额限制），超阈值高亮警告。

#### 4.5.8 审计日志前端

**现状**：`audit_log` 表已建但无查询界面。  
**实现**：管理员在 `/admin/audit` 页面查看审计日志，支持按时间范围、操作用户、操作类型、目标类型筛选，分页展示，点击展开详情（old_value / new_value JSON 对比）。

#### 4.5.9 验收标准

- [ ] **AC5.1**：通知系统在关键事件发生时自动生成通知，前端铃铛实时显示未读计数

## 7. 迭代计划与优先级

### 7.1 优先级定义

| 优先级 | 说明 | 迭代策略 |
|--------|------|---------|
| **P0** | 必须交付，否则迭代2 核心目标无法达成 | 迭代2 前期集中攻克 |
| **P1** | 应该交付，重要但不阻塞核心流程 | 迭代2 中后期完成 |
| **P2** | 可以交付，锦上添花，时间允许则做 | 迭代2 末期或迭代3 前期 |

### 7.2 建议开发顺序

```
第1-2天（P0 攻坚）
├── 需求四：后端 RBAC（@PreAuthorize + SecurityConfig 升级）
├── 需求四：前端路由守卫升级 + 角色菜单过滤
├── 需求三：侧边栏重构（可折叠 + 分组 + 角色过滤）
└── 需求一：工单维修工作台（Tab 结构 + 后端 4 个新 API）

第3-4天（P0 攻坚 + P1 推进）
├── 需求二：数字孪生页面（Three.js 2.5D 场景 + 设备模型 + 状态映射）
├── 需求一：备件快捷申领与工单关联
├── 需求一：工单闭环→生成案例流程
├── 需求三：信息层级重构 + 全局搜索
└── 需求四：MANAGER/SCHEDULER 专属视图

第5-6天（P1 推进 + P2 补充）
├── 需求五：通知系统落地（后端推送 + 前端铃铛 + 通知列表）

```
