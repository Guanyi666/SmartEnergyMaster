| **修复人** | 段冠屹 |

### CR-012 🟠 work_order 表缺失防重复约束导致并发建单

| 项目 | 内容 |
|------|------|
| **发现时间** | 2026-06-12 10:00 |
| **审查人** | 段冠屹（PO） |
| **严重程度** | 🟠 重要（功能性） |
| **所属模块** | 数据库 — 约束设计 |
| **问题描述** | 故障检测引擎在 `SensorDataServiceImpl.analyzeAndTriggerAlarm()` 中使用"先查后插"（SELECT 检查是否存在活跃工单 → INSERT 创建新工单）的幂等逻辑。在并发场景（仿真器 3 秒推送间隔 + 多设备同时故障）下，两个线程可能同时通过 SELECT 检查 → 都执行 INSERT → 创建重复工单 |
| **根因分析** | 应用层"先查后插"不是原子操作，在 READ COMMITTED 隔离级别下存在竞态窗口。需要数据库层的唯一约束兜底 |
| **影响范围** | 同一设备+同一故障类型可能出现多条 PENDING 工单 |
| **修复方案** | 新增部分唯一索引：`CREATE UNIQUE INDEX ix_work_order_unique_active_fault ON work_order (device_id, fault_type) WHERE status IN ('PENDING', 'IN_PROGRESS')`。PostgreSQL 的部分索引（Partial Index）在 WHERE 条件内的行上强制唯一性，从数据库层面彻底杜绝重复活跃工单 |
| **修复人** | 段冠屹 |
| **修复提交** | `11383d0` |
| **复審结果** | ✅ 通过 — 并发建单被数据库拒绝 |

## 审查经验教训

### 做得好的

1. **DDL 幂等补丁模式**：`09_workorder_audit_fixes.sql` 使用 `DO $$ IF NOT EXISTS` 模式，可安全重跑，是迭代开发的良好实践
2. **JsonbTypeHandler 设计**：自定义 TypeHandler 解决了 PG JDBC 驱动的 JSONB 兼容问题，封装清晰，`@MappedTypes` 注解规范
3. **部分唯一索引**：`WHERE status IN ('PENDING', 'IN_PROGRESS')` 的部分索引设计精确表达了"活跃工单唯一"的语义，比应用层防重更可靠
4. **集中式中文映射**：`utils/status.js` 将所有枚举值的中文映射集中管理，避免了散落各处的硬编码翻译

### 需要改进的

1. **合并冲突解决不规范**：GlobalExceptionHandler 被覆盖、路由遗漏、API 导出缺失——三次合并相关问题说明合并时缺少 Checklist。**建议迭代2 引入 PR Review 强制流程（IMP-05）**
2. **DDL 版本管理混乱**：`init.sql` 与 `01_init.sql` 并存、旧文件未清理。**建议迭代2 建立数据库版本号机制或统一到单一 DDL 文件**
3. **并发安全靠应用层**：工单防重和指派防重依赖应用层"先查后插"，在迭代2 引入单元测试之前缺乏并发测试。**数据库层唯一约束应作为应用层防重的兜底**
4. **响应式设计事后补救**：前端开发初期未定义响应式断点规范，在迭代末期发现问题后临时修复。**建议迭代2 开发前制定响应式规范（见《迭代2 需求规格说明书》4.3.3 节）**
