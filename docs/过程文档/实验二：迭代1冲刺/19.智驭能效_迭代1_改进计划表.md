## 三、改进措施详情

### IMP-01：API First 开发流程

**当前问题**：后端接口开发完成后才补 Swagger，前端初期靠口头沟通对接，效率低。

**目标状态**：迭代2启动前，所有新接口的 Swagger 定义（路径、入参 DTO、出参 VO、状态码）100% 完成，前端仅依赖 Swagger 文档开发。

**实施步骤**：
1. 迭代2规划会议中，PO 与后端对齐全部新接口清单
2. 后端在创建 Controller 时同步添加 @Operation + DTO @Schema 注解
3. 前端通过 /swagger-ui.html 查看接口定义，使用 Mock 数据先行开发

**成功标准**：迭代2 站会中无"等待后端 API 文档"类障碍。

### IMP-02：引入单元测试

**当前问题**：只有默认 contextLoads() 测试，全部靠手动验证，回归成本高。

**目标状态**：核心 Service 层至少 30% 行覆盖率，Controller 层至少 2 个 MockMvc 测试。

**实施步骤**：
1. QA 搭建 JUnit 5 + Mockito 测试框架
2. 编写 UserServiceTest（注册/登录/查重）
3. 编写 WorkOrderServiceTest（状态流转/设备联动）
4. 编写 SensorDataServiceTest（故障检测/幂等保护）
5. 配置 JaCoCo 覆盖率报告

**成功标准**：`mvn test` 全部通过，JaCoCo 报告显示核心 Service ≥ 30%。

### IMP-03：技术选型归档

**当前问题**：JWT 双实现选择在开发中才决策，无书面记录。

**目标状态**：CLAUDE.md 架构要点章节明确标注技术选型决策和原因。

**实施步骤**：
1. SM 在 CLAUDE.md 中补充"技术选型决策记录"
2. 记录内容：选型、备选方案、决策原因、决策日期

### IMP-04：设备状态调试接口

**当前问题**：OFFLINE 状态需要等 5 分钟才能验证，测试效率低。

**目标状态**：增加一个调试接口，允许手动触发设备状态变更。

**实施步骤**：
1. 新增 `PUT /api/devices/{id}/status-debug`（仅 dev 环境启用）
2. 支持手动设置 OFFLINE 等状态

### IMP-05：PR Review 流程

**当前问题**：无 Code Review，代码质量依赖个人。

**目标状态**：所有 feature 分支合并到 develop 前，至少 1 人 Review 并 Approve。

**实施步骤**：
1. SM 在 GitHub 仓库配置 branch protection（develop 分支）
2. 制定 Review Checklist（代码规范/安全/性能/测试）
3. 每日分配 Reviewer 轮值

### IMP-06：分析页增强

**当前问题**：分析页仅有基础 24h 曲线，交互功能不足。

**目标状态**：支持时间范围选择、电价时段着色。

**实施步骤**：
1. 增加时间选择器组件（1h / 6h / 24h / 7d）
2. ECharts 图表增加电价时段背景色带（峰=橙/谷=绿/平=蓝）
