## 冲刺目标概述

完成 Spring Boot 3.4.5 + Vue 3 + TimescaleDB 全栈基础架构搭建，实现"**数据采集 → 存储 → 实时监控 → 故障检测 → 工单管理 → 调度建议**"核心业务闭环，交付可独立运行、可演示的系统 MVP 版本。

具体而言，迭代1需达成以下子目标：

1. **数据底座就绪**：TimescaleDB 时序数据库搭建完成，4 张核心业务表（sys_user、device、sensor_data、work_order）就绪，sensor_data 启用超表自动分区
2. **认证体系可用**：Spring Security + JWT 无状态认证完成，支持用户注册/登录/角色权限控制，前端路由守卫生效
3. **设备数字孪生**：3 台钢铁厂核心设备（电弧炉/水泵/空压机）注册在册，支持设备状态自动推导（STOPPED→IDLE→RUNNING→HIGH_LOAD→FAULT→MAINTENANCE→OFFLINE）
4. **数据链路通畅**：Python 仿真器 → POST /api/sensor/upload → TimescaleDB → GET 查询 → 前端大屏，全链路端到端跑通
5. **智能预警可用**：2 条交叉验证规则（机械卡涩/冷却中断）自动触发工单创建，支持工单三态流转（PENDING→IN_PROGRESS→RESOLVED）及设备状态联动
6. **监控大屏可演示**：前端 5 秒轮询实时大屏，含全局指标卡、温度/压力双仪表盘、调度建议、工单面板、设备状态磁贴

## 价值主张

### 解决的核心问题

| 问题 | 当前状态 | 迭代1交付后 |
|------|---------|-------------|
| 设备数据缺乏统一接入和存储 | 无系统 | 仿真器→API→TimescaleDB 标准化数据链路 |
| 设备故障依赖人工巡检发现 | 人工 | 系统自动检测振动/温度异常并生成工单 |
| 能耗状态无法实时掌握 | 无监控 | 大屏 5 秒刷新，指标卡+仪表盘直观展示 |
| 维修流程无数字化管理 | 纸质/Excel | 工单三态流转，状态变更自动联动设备 |
| 分时电价未纳入调度决策 | 人工经验 | 四级调度建议（CRITICAL/WARN/GOOD/INFO）自动生成 |

### 对后续迭代的支撑

- **迭代2（AI引擎）** 需要历史传感器数据作为训练集 → 迭代1的 TimescaleDB + sensor_data 超表提供数据底座
- **迭代3（集成收尾）** 需要稳定的后端 API → 迭代1的 REST API + Swagger 文档提供接口规范
- **全过程** 的演示和答辩 → 迭代1的 Docker 一键部署方案保障可复现

## 目标偏差分析

迭代1 目标达成度远超原计划，主要原因：
1. **团队效率高**：8 位全栈 T 型人才，3 天满载高效协作，实际人效约为计划的 1.3 倍
2. **架构先行策略正确**：Day1 搭建好基础设施（DB/JWT/Cache），Day2-3 上层开发几乎无阻塞
3. **PO 决策及时**：JWT 选型/Swagger 优先/Redis 提前等 5 项关键决策均在 1h 内拍板
4. **SM 进度管控到位**：站会发现的 7 个障碍均当天解决，无积累到第二天

**唯一偏差**：Redis 基础设施从迭代2提前（+15 故事点），虽然增加了范围但缓存显著降低了 DB 压力，属于正向偏差。
