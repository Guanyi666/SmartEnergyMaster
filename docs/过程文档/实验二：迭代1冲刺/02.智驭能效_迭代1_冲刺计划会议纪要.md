## 核心讨论内容与结论

### 迭代规划方案确认

会议讨论了两种迭代规划思路：

**方案一：功能优先**  
先快速开发前端大屏和后端接口，让系统"看起来可用"，再回头补基础设施。

**方案二：架构优先**  
先搭建数据库、认证、核心 API 等基础设施，再在其上构建大屏和业务功能。

**决策：采用"方案二：架构优先"**

决策依据：
1. 系统目标明确——面向钢铁厂的工业能源管理平台，业务需求已在项目计划报告中完整定义，无需通过快速原型验证市场可行性
2. 数据链路复杂度高——"仿真器→后端→数据库→前端大屏"全链路涉及6个技术组件，底层不稳会导致上层全部返工
3. 团队具备全栈能力——Spring Boot 后端、Vue 3 前端、Python 数据管道均可并行推进
4. 项目周期短（10天）——必须一次性搭建稳定基础，后续迭代直接增量开发

### 迭代目标确认

会议一致通过迭代1核心目标：

> **完成 Spring Boot + Vue 3 全栈基础架构搭建，实现"数据采集→存储→监控→告警→调度"核心闭环，交付可独立运行的 MVP 版本。**

核心交付物：
- 可运行的后端 REST API（认证/设备/传感器/工单/仪表盘 5 组接口）
- 可运行的前端 SPA（登录页/监控大屏/分析页/设备管理页 4 个页面）
- Python 设备仿真器（3 设备耦合模型 + 故障注入）
- Docker 一键部署方案

### 待办项澄清与依赖确认

**依赖关系确认：**

1. **数据库 → 后端**：TimescaleDB + init.sql 必须先就绪，后端所有模块依赖数据库
2. **认证 → 其他接口**：SecurityConfig + JWT 必须先通，后续所有受保护接口依赖认证
3. **设备管理 → 传感器上传**：设备必须先注册（deviceCode），传感器上传才能关联设备
4. **后端 API → 前端页面**：后端接口必须提供 Swagger 文档或接口说明，前端才能对接
5. **故障检测 → 工单管理**：故障检测引擎创建的工单通过 WorkOrderService 管理

**并行执行组：**
- 组A（朱靖轩+彭锐豪）：认证 + 设备管理 + 工单管理（可并行）
- 组B（王鑫瑶）：传感器数据 + 故障检测 + 仪表盘（依赖组A的设备管理）
- 组C（李文卓+李文卓）：项目初始化 + 组件开发（用 Mock 数据先行开发，后端就绪后切换）
- 组D（段冠屹）：UCI 数据下载 + 设备仿真器开发（独立，依赖后端上传接口）

**验收标准补充：**
- 所有 API 接口响应时间 ≤ 3 秒（正常负载）
- 前端大屏 5 秒自动刷新，数据与实际一致
- 故障检测覆盖 2 种场景（机械卡涩 + 冷却中断），同一故障不重复创建工单
- 仿真器支持 3 台设备物理耦合、概率性故障注入、24 小时历史回填
- 管理员账号 admin/123456 可登录并访问全部功能
- Docker 部署方案可一键启动 TimescaleDB + 后端

### 风险识别与应对

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| TimescaleDB 超表查询性能不达预期 | 中 | 高 | 预设复合索引，准备降级为普通 Postgres 表 |
| JWT 两套实现（Hutool/jjwt）造成混乱 | 高 | 中 | 明确约定登录签发用 Hutool，备用 jjwt 工具类仅在特殊场景使用 |
| 前端大屏 ECharts 渲染性能不足 | 低 | 中 | 限制同时展示数据点数，超过 500 点启用采样 |
| 仿真器数据与真实数据偏差大 | 中 | 低 | 基于 UCI 真实数据集驱动，物理模型验证过 |
| 团队 3 天周期任务溢出 | 中 | 高 | P2 优先级任务可降级到下个迭代；SM 每日站会跟踪 |
