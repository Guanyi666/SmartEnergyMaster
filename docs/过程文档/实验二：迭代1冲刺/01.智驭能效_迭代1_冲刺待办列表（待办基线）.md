### 迭代背景与总体策略

**项目背景**：智驭能效是面向钢铁厂核心设备（电弧炉、循环水泵、空压机）的工业能源管理平台。系统接收传感器遥测数据，自动检测设备故障并生成维修工单，结合分时电价给出 AI 调度建议。迭代1是项目首个正式开发冲刺，目标是在 3 天内搭建起"数据采集—存储—监控—告警—调度"系统核心链路。

**开发策略选择**：团队在冲刺规划会议上讨论并否决了"功能优先"方案（先开发前端大屏让系统"看起来可用"），一致通过"架构优先"策略，原因如下：
1. **数据链路复杂度高**："仿真器→后端→数据库→前端大屏"全链路涉及 Spring Boot、TimescaleDB、Redis、Vue 3、ECharts、Python 仿真器共 6 个技术组件，底层架构不稳会导致上层全部返工
2. **团队全栈能力互补**：DT 成员兼具前后端开发能力，PO 具有全栈架构经验，SM 具有敏捷管理经验，可并行推进多个模块
3. **10天项目周期紧凑**：Sprint 1 必须一次性搭建稳定基础，Sprint 2 直接增量开发 AI 预测、多 Agent 故障诊断等高级功能
4. **工业系统特性要求**：能源管理系统对数据可靠性和实时性要求高，基础架构质量直接决定后续功能可用性

### 冲刺目标

**总目标**：完成 Spring Boot + Vue 3 全栈基础架构搭建，实现"设备数据采集 → 时序存储 → 实时监控大屏 → 静态阈值异常检测 → 维修工单管理 → 仪表盘聚合总览"全链路闭环，并通过 Docker Compose 实现一键部署，交付可独立运行的系统 MVP（最小可行产品）版本。

**目标分解**：

| 目标维度 | 具体目标 | 衡量标准 |
|---------|---------|---------|
| 后端架构 | Spring Boot 3.4.5 + MyBatis-Plus + Spring Security + JWT + TimescaleDB | 5 组 Controller 全部可用，Swagger 文档在线 |
| 前端架构 | Vue 3 + Vite + Element Plus + ECharts + Pinia | 4 个页面可交互，5s 轮询正常 |
| 数据管道 | Python 设备仿真器（3 设备耦合模型 + 故障注入） | 持续推送真实数据到后端，故障率符合预期 |
| 部署方案 | Docker Compose（TimescaleDB + Redis + 后端） | `docker-compose up -d` 一键启动 |
| 数据链路 | 仿真器→后端→数据库→前端全链路 | 端到端延迟 ≤ 3s，数据一致性正确 |

### 完成定义（DoD）

每项待办任务必须满足以下全部条件方可标记为"已完成"：

1. **代码完成**：功能代码已提交至 GitHub `develop` 分支，通过至少 1 人 Code Review，无未解决的 Review 意见
2. **编译通过**：`mvn clean compile` 无错误（后端），`npm run build` 无错误（前端），Python 脚本语法正确
3. **测试通过**：核心功能通过手动冒烟测试，无阻塞性（P0/P1）缺陷；已发现的 P2 缺陷已记录在缺陷跟踪表中
4. **可演示**：功能在前端可正常运行和展示，核心用户旅程（登录→查看大屏→切换设备→处理工单）可完整走通
5. **文档同步**：API 接口已添加 `@Operation` +参数 `@Parameter` 注解，关键业务逻辑有清晰的注释说明
6. **集成验证**：数据链路（仿真器→后端→数据库→前端）可端到端跑通，数据在各环节传递正确且延迟可接受
7. **安全检查**：敏感接口（除白名单外）均受 JWT 保护，无硬编码密钥提交到代码仓库

### 任务依赖关系图

基于项目架构，迭代1任务之间存在以下关键依赖链：

```
环境搭建（S1.1.1）
    ├──→ 数据库建表（S1.2.1, S1.2.2）
    │       └──→ 后端 CRUD（S2.3.x, S2.4.x, S2.6.x）

    │       ├──→ 认证模块（S2.2.x）
    │       │       └──→ 所有受保护接口的前端对接
    │       ├──→ 设备管理（S2.3.x）
    │       │       └──→ 传感器上传（S2.4.1）需要 device_id
    │       └──→ 故障检测（S2.5.x）
    │               └──→ 工单管理（S2.6.x）
    ├──→ 前端项目初始化（S3.1.x）
    │       └──→ 所有前端页面（S3.2.x - S3.5.x）
    └──→ Docker 部署（S1.3.1）
```

**关键路径（Critical Path）**：环境搭建 → 数据库 → 后端初始化 → 认证模块 → 设备管理 → 传感器上传 → 故障检测 → 工单管理 → 仪表盘 → 前端大屏。任何关键路径上的延迟将直接影响迭代交付。
