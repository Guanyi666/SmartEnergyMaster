## 障碍详情

### IMP-001：JWT 双实现选择混乱

| 项目 | 内容 |
|------|------|
| 发现人 | 彭锐豪 |
| 发现时间 | 6/8 下午 站会 |
| 描述 | 项目中存在两套 JWT 实现——Hutool JWT（用于 Filter 校验）和 jjwt（JwtUtils 工具类）。彭锐豪 在开发认证模块时不确定应该用哪套实现 |
| 影响 | 认证模块开发暂停 30 分钟 |
| 解决方案 | PO 决定：登录签发用 Hutool JWT，jjwt 工具类保留备用。在 CLAUDE.md 中明确标注 |
| 经验教训 | 技术选型应在迭代启动前明确，避免开发中决策 |

### IMP-002：前端等待后端 API

| 项目 | 内容 |
|------|------|
| 发现人 | 李文卓 |
| 发现时间 | 6/8 下午 站会 |
| 描述 | 前端大屏开发需要后端 API 数据，但后端 API 尚未就绪 |
| 影响 | 前端开发进度可能延迟半天 |
| 解决方案 | SM 建议用 Mock 数据先行开发组件，API 就绪后切换。李文卓 在 api/http.js 中预留接口路径，用 JSON 文件模拟返回 |
| 经验教训 | 接口文档（至少接口路径+返回结构）应在开发前对齐 |

### IMP-003：后端 API 文档不完整

| 项目 | 内容 |
|------|------|
| 发现人 | 李文卓 |
| 发现时间 | 6/9 上午 站会 |
| 描述 | 后端接口缺少 Swagger 注解，前端不知道入参字段名、返回结构 |
| 影响 | 前端联调效率低，需要反复沟通确认 |
| 解决方案 | 朱靖轩/2/3 紧急补充 @Operation 注解和 DTO/VO 的 @Schema 注解；同时在微信群里发 curl 示例 |
| 经验教训 | Swagger 应在开发阶段同步配置，不应等到迭代末补 |

### IMP-004：仿真器物理模型参数调优

| 项目 | 内容 |
|------|------|
| 发现人 | 数据 |
| 发现时间 | 6/9 上午 |
| 描述 | 仿真器中的炉膛热惯性系数、轴承磨损率、冷却健康度衰减率等物理参数需要根据真实数据校准，但 UCI 数据集不直接提供这些参数 |
| 影响 | 仿真数据可能与真实设备行为偏差较大 |
| 解决方案 | 基于 UCI 数据集的 usage_kwh/温度/振动范围反推参数初始值，在代码注释中标注"参数基于 UCI 数据集统计反推，待真实设备数据校准" |
| 经验教训 | 仿真器参数应建立参考基准文档，便于后续校准 |

### IMP-005~007

已在缺陷跟踪记录（BUG-001/002/004）中详细记录，此处不再重复。

## 障碍分析

| 维度 | 数量 | 占比 |
|------|------|------|
| 技术决策类 | 1（IMP-001） | 14% |
| 前后端协作类 | 2（IMP-002/003） | 29% |
| 开发工具/环境类 | 1（IMP-007） | 14% |
| 功能缺陷类 | 3（IMP-004/005/006） | 43% |

**主要发现**：
1. **前后端协作类障碍占比最高（29%）**，反映"API First"原则在迭代1执行不力。后端应先在 Swagger 定义接口，前端基于 Swagger 文档开发，而非口头沟通
2. **技术决策类障碍（14%）** 虽只一项但影响面大——JWT 双实现混乱消耗了3个仓库贡献者的时间和精力
3. **功能缺陷类障碍（43%）** 虽然比例最高但多为开发期自测发现，属于正常开发过程的"摩擦成本"
