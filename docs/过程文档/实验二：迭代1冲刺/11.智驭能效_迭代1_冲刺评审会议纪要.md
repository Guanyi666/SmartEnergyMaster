## 评审过程中的技术讨论

### 讨论1：故障检测规则的扩展路径

**背景**：当前仅 2 条静态阈值规则（机械卡涩、冷却中断），指导老师询问扩展计划。

**讨论要点**：
- 段冠屹介绍了迭代2 的计划——引入 LLM 多 Agent 故障诊断系统，通过 5 个 Agent 协作（数据预处理 Agent、异常检测 Agent、根因推理 Agent、维修建议 Agent、报告生成 Agent）实现智能诊断
- 指导老师建议在引入 LLM 前先增加基于统计的动态基线（滑动窗口均值 ± 3σ），作为 LLM 的"预筛器"
- 结论：迭代2 同时推进动态基线（规则引擎升级）和多 Agent 系统

### 讨论2：系统实时性能的工业级要求

**背景**：指导老师提问"钢铁厂真实场景中 PLC 数据采集频率通常是 100ms-1s 级别，当前 3s 仿真周期是否足够？"

**讨论要点**：
- 当前仿真器 3s 周期是为 UCI 数据集（15min 间隔）的演示考虑
- 真实工业场景中系统需要支持更高的采集频率
- TimescaleDB 超表设计支持高吞吐写入（单节点可到 1M+ metrics/s）
- 结论：当前架构支持更高频率（通过减小仿真器推送间隔即可），但需要在真实环境中做压力验证

### 讨论3：安全性审查

**背景**：指导老师询问系统的安全防护措施。

**讨论要点**：
- JWT + Spring Security 保护所有敏感接口，白名单仅 4 条路径（auth + sensor 上传/查询）
- 登录限流：同一 IP 1 分钟内最多 3 次失败尝试（Redis 滑动窗口）
- 密码 BCrypt 加密存储
- CSRF 已禁用（SPA 无状态架构）
- 指导老师建议补充：API 请求日志审计、生产环境 JWT 密钥更换机制
- 结论：迭代2 补充请求日志，迭代3 补安全审计
