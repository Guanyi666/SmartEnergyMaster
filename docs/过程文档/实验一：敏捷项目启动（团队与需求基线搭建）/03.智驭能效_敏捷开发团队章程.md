## 章程总则

### 制定目的

为明确敏捷开发团队的核心目标、角色职责、协作规范及运作机制，强化团队自组织能力，保障迭代交付效率与成果质量，构建透明、协作、持续改进的团队文化，特制定本章程。本章程作为团队内部协作的核心准则，全体成员需共同遵守并践行。

### 适用范围

本章程适用于智驭能效（SmartEnergyMaster）项目的所有团队成员，涵盖项目启动、迭代开发、交付评审及收尾复盘全流程。

### 核心目标

- 以用户需求为核心，通过迭代冲刺实现"小步快跑、快速交付"，确保输出符合工业能源管理场景的高质量成果；
- 建立高效协作机制，减少沟通壁垒，提升团队整体开发效率；
- 培养团队成员的敏捷思维与自组织能力，实现持续改进；
- 顺利完成课程实验项目的各项迭代任务与交付目标；
- 交付一套贴近真实钢铁厂使用场景的智慧能源管理平台。

## 核心角色与职责

团队采用敏捷Scrum框架角色划分，实行自组织管理，各角色分工协作、责任共担。

### 产品负责人（Product Owner, PO）

- **需求管理**：主导需求分析，梳理"设备监控大屏""传感器数据采集""AI故障诊断""智能调度优化"等用户故事，构建用户故事地图与产品待办列表（Product Backlog），明确"基础数据采集→实时监控→智能告警→AI诊断→优化调度"的需求优先级；
- **目标对齐**：向团队传递项目核心价值与需求背景，确保团队开发方向与工业能源管理场景一致；
- **决策担当**：负责迭代待办列表（Sprint Backlog）的优先级确认，迭代成果的验收决策；
- **沟通桥梁**：对接指导老师，同步需求变更与反馈，向团队传递准确信息。

### 敏捷教练（Scrum Master, SM）

- **流程保障**：熟悉敏捷Scrum流程，引导团队严格执行迭代规划、每日站会、迭代评审、迭代回顾等核心会议；
- **障碍清除**：主动收集团队开发过程中的障碍（LLM API集成、Redis配置、TimescaleDB超表查询等技术难题），协调资源协助解决，确保迭代顺利推进；
- **团队赋能**：引导团队自组织运作，鼓励成员主动承担责任，培养团队的协作意识与问题解决能力；
- **文档统筹**：督促各环节输出规范材料（燃尽图、站会记录等），确保实验过程可追溯。

### 开发团队（Development Team, DT）

开发团队为跨职能自组织团队，成员共同承担前端、后端、AI模型、部署等任务，无层级划分，具体职责包括：

- **任务执行**：根据迭代待办列表，自组织拆分"后端API开发""前端大屏开发""AI模型训练""Agent系统搭建"等任务并落实，按时完成迭代开发与测试工作；
- **质量把控**：严格遵守代码规范，编写单元测试，确保交付成果符合"接口响应≤500ms""大屏刷新周期≤5s""AI诊断响应≤30s"等非功能需求标准；
- **主动协作**：在每日站会同步个人进度、遇到的问题及计划，主动协助团队成员解决技术难题；
- **参与改进**：积极参与迭代回顾会议，提出流程优化建议，推动团队持续改进。

## 团队价值观与行为准则

### 核心价值观

- **用户导向**：始终以工业场景真实需求为核心，拒绝无价值的开发工作；
- **透明开放**：工作进度、问题障碍、风险隐患及时同步，不隐瞒、不推诿；
- **协作共赢**：打破个人壁垒，主动分享经验与资源，以团队目标为首要目标；
- **持续改进**：正视问题与不足，通过迭代回顾总结经验，不断优化工作方式。

### 行为准则

- **准时参与**：所有团队会议需准时参加，若无法参与需提前向SM及团队请假；
- **主动担当**：认领任务后需全力落实，若遇困难及时反馈，不拖延、不敷衍；
- **尊重沟通**：沟通时清晰表达观点，耐心倾听他人意见，理性讨论分歧，避免情绪化争执；
- **成果负责**：对个人参与开发的成果质量负责，主动配合测试与修改工作；
- **代码规范**：统一采用驼峰命名、RESTful API设计、Git语义化提交信息（feat/fix/docs/refactor）。

## 协作与沟通机制

### 会议规范

| 会议类型 | 时间 | 议程要点 | 产出 |
|----------|------|----------|------|
| 每日站会 | 上午 9:30，≤15min | 昨日进展、今日计划、当前障碍 | 站会纪要 |
| 迭代规划 | 迭代首日 | 需求梳理、任务拆分、工作量估算 | Sprint Backlog |
| 迭代评审 | 迭代末日 | 成果演示、验收反馈 | 评审记录 |
| 迭代回顾 | 评审后 1h 内 | 改进识别、行动项制定 | 回顾纪要 + 改进计划 |

### 沟通工具与渠道

- **即时沟通**：采用微信工作群作为日常沟通工具，用于同步"接口开发完成"等紧急信息、协调跨角色任务衔接；
- **代码协作**：使用 GitHub 托管代码，main 主分支 + develop 开发分支 + feature/xxx 功能分支，每完成一个功能模块及时提交代码并撰写清晰的 commit message；
- **文档协作**：项目文档（Markdown格式）存放于 docs/ 目录，纳入 Git 版本控制；
- **进度可视化**：通过 GitHub Projects 看板实时更新任务状态（待办、进行中、已完成、阻塞），每个任务卡片标注负责人与预计完成时间。

## 迭代管理规范

### 迭代周期与规划

- **迭代周期**：项目分为3轮迭代，每轮迭代周期为2-3天，总计10天。
  - 第一轮（第3-5天）：MVP交付——设备数据采集 + 监控大屏 + 基础告警 + Swagger文档 + Redis基础设施
  - 第二轮（第6-7天）：核心智能化——LLM故障诊断 + 多Agent系统 + 人员管理 + AI能耗预测
  - 第三轮（第8-9天）：业务闭环——前端全面升级 + 行业标准功能 + 部署联调
- **任务估算**：采用"故事点"进行任务工作量估算（1点=约2小时），经开发团队共同讨论确定；
- **迭代目标**：每轮迭代需明确可衡量的目标，避免目标模糊。

### 开发与交付规范

- **代码规范**：后端遵循 Java 驼峰命名，RESTful API 以 `/api/功能模块/操作` 格式命名；前端采用 Vue 3 Composition API + 语义化组件命名；Python 遵循 PEP 8；
- **提交规范**：`type(scope): message`，如 `feat(backend): add device health score API`；
- **交付物要求**：每轮迭代交付物需包含可运行的程序包、轻量测试报告、简易用户操作说明，确保"可演示、可使用"；
- **变更管理**：迭代期间若需变更需求，需经PO评估优先级，团队讨论影响后决定是否纳入当前迭代，优先放回产品待办列表待后续规划。

### 风险与问题处理

- **风险预判**：迭代规划阶段同步识别潜在风险（如"LLM API调用延迟""Redis集群配置不熟悉"等），制定应对方案；
- **问题上报**：成员遇到无法独立解决的问题时，需在当日站会或即时沟通中上报，由SM协调解决；
- **记录跟踪**：建立"风险与问题跟踪表"，记录问题描述、处理责任人、解决进度及结果，确保闭环。

## 团队成长与改进机制

### 能力提升

- **知识分享**：每周安排一次20分钟技术分享，内容涵盖"Spring Security JWT认证机制""Vue 3 Composition API最佳实践""LSTM时序预测原理""LLM Prompt Engineering"等；
- **代码评审**：关键模块代码提交前需至少1人Review，互相学习提升。

### 持续改进

- **改进落地**：迭代回顾会议提出的改进措施，需明确责任人与完成时间，纳入下一轮迭代待办列表；
- **章程修订**：每轮迭代结束后，团队可根据实际运作情况，共同讨论修订本章程，确保章程贴合团队需求。
