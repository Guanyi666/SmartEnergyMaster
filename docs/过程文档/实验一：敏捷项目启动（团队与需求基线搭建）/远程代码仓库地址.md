## 团队角色与 GitHub 权限映射

| **团队角色**             | **人数** | **建议 GitHub 权限**  | **核心权限说明**                                             |
| ------------------------ | -------- | --------------------- | ------------------------------------------------------------ |
| **PO **                  | 1        | **Admin** (管理员)    | 拥有仓库的最高权限，可修改仓库设置、配置集成工具、管理团队成员权限。 |
| **SM (Scrum Master)**    | 1        | **Maintain** (维护者) | 协助管理仓库。可以创建/管理 Label、Milestone、管理 Issue 和 PR，但无法修改高风险的仓库设置（如删除仓库）。 |
| **Developer (开发人员)** | 6        | **Write** (写入者)    | 拥有代码推拉权限。可以创建分支、提交代码、发起 PR。**但不能直接推送到受保护的分支**。 |

## 分支管理与保护策略

不允许任何人（包括开发人员）直接 `git push` 到主分支。启用 GitHub 的 **Branch Protection Rules**（分支保护规则）。

### 针对 `main` 分支的保护设置：

1. **Require a pull request before merging（合并前必须经过 PR）**：
   - 所有代码必须通过 Pull Request 才能合并入主分支。
2. **Require approvals（必须经过代码审查）**：
   - 设置 **Required approving reviews = 1 或 2**。
   - **日常规范**：开发人员提交 PR 后，必须由至少一名其他开发人员（Peer Review）或 SM/PO 审核通过后，才能合并。
3. **限制推送权限**：
   - 确保没有任何人能绕过 PR 直接 push。

## 日常敏捷开发协作工作流

组内遵循以下规范：

```
[Issue 任务分配] ➔ [创建 Feature 分支] ➔ [本地开发与 Push] ➔ [发起 PR] ➔ [代码评审/测试] ➔ [合并入主分支]
```

- **Issue 管理**：由 PO和 SM 共同维护。PO 负责明确需求（User Story），SM 负责组织开发人员认领 Issue 或分配任务。
- **分支命名规范**：开发人员在 `Write` 权限下，自行创建分支，命名为 `feature/issueID-description`（例如 `feature/12-add-timescale-pump`）或 `bugfix/issueID-description` 或 dev/name。
- **PR 合并权限**：
  - 开发人员自己**无权**在未经 Review 的情况下合并自己的 PR。
  - 由 **SM** 或**被指定的 Reviewer** 在确认代码无误、测试通过后，执行最终的 `Merge pull request`。
